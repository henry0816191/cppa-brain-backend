Discussion on the potential inclusion of a null-terminated string view type in Boost, its design considerations, and comparison with existing and proposed solutions.
Key Points (7)
  • The primary motivation for a null-terminator guaranteed string view type is to avoid common pitfalls of `std::string_view`, where `data()` is not guaranteed to be null-terminated and `size()` does not account for it. Such a type would simplify interactions with C APIs and prevent unnecessary conversions to `std::string`. [1] [2]
  • Several existing and proposed solutions for a null-terminated string view were discussed, including Chromium's `cstring_view.h`, C++ standardization proposals (e.g., P3655), and Rust's `CStr`. Comparing these implementations and their divergences is considered important. [3] [4] [2]
  • There's a debate on the extent of functionality a null-terminated string view should offer. Some argue it should primarily provide C-string like operations (`data`, `operator[]`, `length`/`strlen`) without the full `std::string` interface to avoid over-engineering, while others suggest aligning it with `std::string_view`'s interface for consistency, which implies methods like `end()` and `substr`. [5] [6] [7]
  • Key desired features for a new type include distinguishing between a null-terminated sequence and a raw `char*` pointer, providing shorter syntax than `.c_str()` for `std::string` objects, and ensuring `operator==` compares content rather than addresses. Runtime invariant checking is also seen as a benefit. [5]
  • The performance impact of `strlen` for calculating length is debated. While some note that compilers can optimize away `strlen` calls for inlined static strings, others emphasize that `strlen` is performant for typical string lengths and essential for operations beyond just size. [5] [2] [6]
  • The fundamental incompatibility between `std::string_view`'s substring guarantee (no memory management) and a null-terminator guarantee is acknowledged. The idea of using a null-terminator-detecting sentinel for range iteration is mentioned but noted as deviating from a contiguous range. [5]
  • The discussion touches on the evolution of `std::string_view`'s interface in C++23, which accepts contiguous sized ranges, and how this differs from earlier versions that relied solely on `data()` and `size()`. A proposed `string_like` concept aims to extend compatibility for `cstring_view` similarly. [2] [7] [7]

Discussions on streamlining Boost release archives by removing the unified documentation build and splitting into source-only and full archives.
Key Points (5)
  • The primary goal is to streamline the Boost release package and website. Two main ideas are proposed: removing the unified documentation build and splitting the release into separate source and documentation archives. The unified documentation build has a non-standard layout that complicates tooling, while separate archives would offer faster downloads for users who don't need local documentation. [25] [26] [27] [28] [29] [30]
  • Removing the unified documentation build involves moving documentation into individual library directories by editing Jamfiles. This task requires effort from each library maintainer and is considered a worthwhile improvement for consistency and modularity. However, the actual effort for the root /doc/ tree build is minimal, primarily involving updates to the top-level redirect HTML file. [31] [26] [32]
  • Regarding the split archives, the 'full' archive would remain the primary URL for backwards compatibility, while an alternative 'source-only' archive would be offered for faster downloads. The contents of these archives, particularly whether to include library-specific HTML documentation files in source archives, are open for discussion. [25] [26] [30] [33]
  • Unofficial source-only release archives (e.g., 'b2-nodocs') have been available on GitHub releases, prepared by a CI job. These archives are roughly half the size of their 'cmake' counterparts because the 'doc/' subdirectories are deleted. The need for separate b2 vs. cmake archives in these unofficial releases was questioned. [27] [28] [29] [33] [33]
  • The decision to proceed with streamlining is acknowledged as being 'on the radar' but also subject to other ongoing tasks. The use of GitHub releases as an alternative to the main website download links was also suggested. [26] [34]

Discussion on potential graduation projects for senior students involving Boost libraries, including a proposal for concurrent set/map implementations.
Key Points (5)
  • The initial inquiry was about identifying Boost-related projects suitable for senior students' graduation work. This led to a suggestion for students to work on implementing concurrent set, multiset, map, and multimap structures based on red-black trees enhanced with positional access. [57] [58]
  • The proposed concurrent data structures offer O(log N) time complexity for insertions and deletions, even for non-extremal elements, differentiating them from concurrent vectors. They aim for 100% STL compatibility and utilize a spinlock for managing concurrent reads and writes. [58]
  • The project for students is divided into two levels: a simpler task involving replacing the current mutex with a modern one and designing parallel algorithms for tree destruction and copying, and a more complex level that includes improving the balancing algorithms. [58]
  • The proposer believes these concurrent container implementations could be of interest to major C++ compiler vendors as they may not currently offer such algorithms. The existing code is available on GitHub, and the proposer is willing to provide optimized spinlock code. [58]
  • There was a consensus that mentors should create a database of tasks, categorized by required knowledge levels, to effectively match students with suitable projects. Using GitHub with specific labels for Boost libraries was suggested as a platform for this database. [58] [58]
